# Масштабирование и несколько экземпляров сервиса

Коротко: не все сервисы в этом репозитории одинаково безопасно запускать в
нескольких экземплярах. Ниже — по каждому, можно или нет, и почему.

## Можно свободно масштабировать

### `nx-worker-stub`
Конкурирующие consumer'ы на `jobs.process`/`jobs.validate` — штатный паттерн
RabbitMQ (round-robin между всеми, кто подписан на одну и ту же очередь).
`prefetch=1` (`WORKER_PREFETCH`) гарантирует, что ни один инстанс не
«нахватает» больше одного задания, пока не подтвердит предыдущее.

```bash
docker compose up -d --scale nx-worker-stub=3
```

Единственное, что стоит держать в голове: `license-server-stub` со своим
пулом мест (`LICENSE_SEATS`) — общий для всех реплик воркера. Больше
воркеров не значит больше параллельных лицензий; они будут конкурировать за
один и тот же пул через `/checkout`.

### `file-storage-service`, `auth-stub`, `license-server-stub`
Stateless-запросы (файлы читаются/пишутся по ключу на общем volume,
проверки не хранят состояние между запросами — кроме `license-server-stub`,
у которого состояние **в памяти процесса**, см. ниже) — можно ставить за
балансировщик без проблем, если сами понимаете следствие ниже.

**Важная оговорка про `license-server-stub`**: пул мест — in-memory,
свой у каждого инстанса. Два реплики `license-server-stub` = два независимых
пула по `LICENSE_SEATS` каждый, а не один пул на двоих. Пока это Phase 0
заглушка, масштабировать её ВООБЩЕ НЕ НУЖНО (один инстанс), пока пул не
переедет во что-то общее (Redis/Postgres) — иначе ограничение на число
лицензий перестанет что-либо ограничивать.

## Пока нельзя масштабировать — и это проверено вживую, не гипотеза

### `job-state-service`
**Держите ровно один инстанс.** Он подписан на именованные (не эксклюзивные)
очереди `jobs.job-state-service.intake` и `jobs.job-state-service.completions` — если
запустить второй инстанс, RabbitMQ поделит между ними сообщения по
round-robin, как для любых конкурирующих consumer'ов. Проблема в том, что
`intake` (создание задачи в БД) и `completions` (обновление статуса) для
ОДНОГО И ТОГО ЖЕ `job_id` могут достаться РАЗНЫМ инстансам, и если у них
разные подключения к БД (например, кто-то по ошибке смотрит в другую базу),
события навсегда разъезжаются. Мы поймали это ровно так на практике: два
случайно оставшихся живыми контейнера `job-state-service` привели к
бесконечным `"completion for unknown job"` в логе одного из них.

Если реально нужно горизонтальное масштабирование `job-state-service` —
это отдельная задача (Phase 1+): либо leader election (активен только один
инстанс, остальные в hot standby), либо переход на другую топологию очередей
(например, consistent hashing по `job_id` вместо round-robin), либо перенос
retry-состояния из памяти процесса в БД, чтобы оно не зависело от того,
какой именно инстанс получил сообщение.

## Как проверить, что у вас случайно не запущено два экземпляра

Симптом из практики: в логе `job-state-service` крутятся
`"completion for unknown job ... requeueing"`, а `jobs`/`job_attempts` в БД
пустые или неполные, хотя `api-server` точно публикует события.

```bash
# Docker Compose
docker compose ps job-state-service          # должна быть 1 строка

# Kubernetes
kubectl get pods -l app=job-state-service    # должен быть 1 Running под
```

В логах `job-state-service` есть диагностика на этот случай "из коробки":
строки `pid=...` в `connected to PostgreSQL` / `persisted job` /
`diagnostic: row for ... seen by pid=...` — если в одном инстансе видите
разные PID для "должно было быть записано" и "не вижу эту строку", это
стопроцентный признак двух живых процессов на одних очередях.

## restart-политики

Всем сервисам в `docker-compose.yml` выставлен `restart: unless-stopped` —
упавший контейнер поднимается сам, без ручного вмешательства. В Kubernetes
эквивалент — `restartPolicy: Always` (значение по умолчанию для Pod) плюс
`livenessProbe`/`readinessProbe`, указывающие на `/health` каждого сервиса
(эти эндпоинты уже реально отражают готовность зависимостей — RabbitMQ,
Postgres — а не всегда отвечают "healthy", как было в исходных заглушках).
