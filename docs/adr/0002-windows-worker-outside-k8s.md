# 0002 — Windows NX-воркер вне Kubernetes-кластера

Дата: 2026-07-16
Статус: принято

## Контекст

Сервер NX доступен только под Windows. Windows-контейнеры в Kubernetes
технически возможны (containerd + Windows Server Core), но:
- значительно тяжелее и дольше стартуют, чем Linux-контейнеры;
- часть managed Kubernetes (EKS/AKS/GKE) поддерживает Windows-ноды с
  ограничениями (не весь набор системных DaemonSet работает, сетевые
  плагины ограничены);
- лицензирование NX (FlexLM) может накладывать ограничения на запуск в
  контейнере/VM — требует проверки условий лицензии.

## Решение

NX-воркер разворачивается как отдельный процесс/служба на Windows
VM/сервере вне Kubernetes-кластера. Воркер — обычный AMQP-consumer
RabbitMQ, обращается к `file-storage-service` и `license-server` по сети
(HTTP/SMB). Масштабирование пула воркеров — вручную или отдельным
скриптом/агентом, не через k8s-примитивы.

## Последствия

- Нет единого механизма самовосстановления/автоскейлинга воркеров через
  k8s — нужно закладывать это отдельно (Phase 3 в ROADMAP).
- Наблюдаемость (логи/метрики) воркера должна быть организована отдельно
  от остального k8s-стека (тот же `log-collector-stub`, но по сети).
