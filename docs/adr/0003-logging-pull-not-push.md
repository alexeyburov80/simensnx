# 0003. Логирование: pull-модель (Promtail/Loki), а не push-сервис log-collector

## Статус
Принято.

## Контекст
В Phase 0 ROADMAP.md был запланирован `log-collector-stub` — сервис, в
который остальные компоненты должны были бы отправлять логи по сети
(push-модель). По мере того как сервисы обзавелись реальной логикой, стало
ясно, что развивать `log-collector-stub` дальше как push-приёмник означало
бы решать в собственном коде задачи, которые уже давно решены готовыми
инструментами: буферизацию при недоступности приёмника, retry с бэкоффом,
backpressure, гарантии доставки. Мы уже один раз проходили этот путь для
интеграции с RabbitMQ (heartbeat, автопереподключение) — повторять его же
для логирования не имеет смысла.

## Решение
Логи каждого сервиса пишутся туда, куда они и так уже пишутся — в
`stdout`/`stderr` контейнера (`qInfo()`/`qWarning()`). Забор и агрегация —
задача инфраструктуры, а не сервисов:

- **Promtail** читает логи всех контейнеров напрямую с диска (через Docker
  service discovery по `/var/run/docker.sock`), без каких-либо изменений в
  коде сервисов и без отдельного шага "подключить логирование" для новых
  сервисов.
- **Loki** хранит и индексирует логи по лейблам (`service`, `container`,
  `stream`), без полнотекстового индекса — дёшево по ресурсам.
- **Grafana** — единая точка просмотра логов (и в перспективе метрик
  Prometheus в том же UI).

`log-collector-stub` удалён из `docker-compose.yml`, Helm-чартов и
`ARCHITECTURE.md`.

## Последствия
- Сервисам никогда не нужно знать адрес лог-приёмника — на один меньше
  сетевой зависимости в бизнес-логике.
- Добавление логирования новому сервису = ничего специального делать не
  нужно, docker-compose service discovery подхватит его автоматически.
- В Kubernetes прямой аналог того же подхода — Promtail/Fluent Bit как
  DaemonSet, читающий `/var/log/pods` с каждой ноды; переход с
  docker-compose не потребует менять эту архитектурную идею.
- Если в будущем понадобится push от нестандартного источника (например,
  NX-воркер на Windows, вне docker/k8s, где такого service discovery нет),
  для него это будет осознанное отдельное решение (Promtail в режиме agent
  на Windows-хосте, или Grafana Agent), а не переиспользование выброшенной
  заглушки.
